# Phase 3: MCP 结果处理与代码写入

收到 MCP 返回后，**直接执行**代码生成与写入，不调用子代理。

---

## 关键理解

**MCP 工具不会生成最终代码，也不会写入任何文件。MCP 只返回"指令和素材"，所有代码生成和文件写入 100% 由CodeBuddy完成。**

具体来说：
- **`order` 字段**：是执行指令——包含代码模板规范、工作流模式、目标路径、约束条件。你必须**遵循这些指令**去生成代码。
- **`code` 字段**：是你的输入素材——可能是带 TODO 的半成品代码（高分用例），也可能是纯文字的用例描述。你必须**基于这些素材**生成完整的可执行代码。
- **MCP 不会执行任何后续动作**。`order` 中出现"使用 edit_file 工具写入"等措辞，也是在告诉**你**去执行写入操作，不是 MCP 自己去做。

**你在本阶段的唯一目标：读取 MCP 返回 → 理解指令和素材 → 生成完整代码 → 写入文件 → 验证文件。**

---

## MCP 返回结构解读（收到返回后必须先读此章节！）

MCP `generate_code` 根据调用模式不同，有两种返回结构：

### 返回结构 A：node 模式首次调用

返回 `uuid_list`（UUID 数组）+ `trace_id`。这**不是最终结果**，需要用返回的 UUID 列表按批次继续调用（进入步骤4批量处理）。

### 返回结构 B：uuid / text_case 模式调用

返回以下 4 个字段：

| 字段 | 含义 | 你要做什么 |
|------|------|-----------|
| `status` | `"success"` 表示 MCP **正常返回**（ 不代表 code 里一定是代码！）| 检查是否为 `"success"` |
| `order` | 一段完整的指令文本（可能非常长），包含代码模板规范、工作流模式、目标路径等 | 提取模板规范、工作流模式、目标路径 |
| `code` | **关键分支点**，有两种正常形态（见下表）| 根据形态决定处理方式 |
| `trace_id` | 追踪 ID | 记录，用于后续上传 |

### `code` 字段的三种正常形态（都必须全力去生成代码）

| 形态 | 识别特征 | 含义 | 你的动作 |
|------|---------|------|---------|
| **代码框架**（高分用例） | 含 `def test_`、`import`、`#TODO` 等 Python 代码结构 | MCP 已生成骨架代码 | `尽全力`补全所有 `#TODO` 处的代码 |
| **用例描述**（纯文本用例） | 含 `用例名:`、`步骤1:`、`预期结果1:` 等描述性文字 | MCP 未生成代码骨架（**这是正常的！不是错误！**）| 基于描述 + `order` 模板 + 项目代码，**尽全力从零生成完整测试代码** |
| **用户描述**（非结构化输入） | `code` 是用户的测试描述文本（非代码、非结构化步骤），`order` 是用户的完整问题，`order` 包含 `code` 的内容以及更多上下文 | **这是正常的！不是错误！** | 综合 `order`（完整问题）和 `code`（测试描述）理解用户意图，结合项目代码自行搜索相关实现并从零生成完整测试代码。此时没有 MCP 提供的模板规范，参考项目中已有的测试代码风格 |

> **特别强调**：以上三种形态都是 `status: "success"` 的正常返回，绝对不是 MCP 内部错误！遇到这些情况**不要**走规则5的错误处理路径，必须继续完成代码生成。

## MCP 返回字段说明

| 字段 | 说明 | 你要做什么 |
|------|------|------------|
| `order` | 包含：1) **代码模板规范**（优先级最高）2) 生成指令（模式、工作流、路径、约束） | 解析并遵循其中的模板和指令去生成代码 |
| `code` | 高分用例：带 `#TODO` 的半成品代码；低分用例：用例描述文本 | 作为输入素材，生成完整代码 |
| `status` | 执行状态 | 检查是否成功 |
| `trace_id` | 追踪 ID | 记录，用于后续上传 |

### 关键规则

1. **目标路径以已确认的生成路径策略为准**：如果 Phase 1 / Phase 2 已确认用户指定生成路径，则优先使用用户指定路径；如果用户未指定，则使用 MCP 返回的 `file_path`
2. **代码模板以 `order` 为准**：`order` 字段开头包含的模板规范具有最高优先级，生成代码必须严格遵循该模板，禁止因"与项目已有代码风格一致"而偏离

## 执行流程

### 第1步：解析 MCP 返回

从 `order` 字段中提取以下信息（它们是你后续生成代码和写入文件的依据）：
- **模式**（generate）
- **工作流**（file_append / create_file / dir_append）
- **目标路径**（file_path）→ 这是你要写入的文件/目录路径
- **代码模板规范**（order 开头部分）→ 这是你生成代码必须遵循的格式
- **约束条件**

### 第2步：获取用户名

```bash
USER_NAME=$(git config user.name 2>/dev/null || true)
if [ -z "$USER_NAME" ]; then USER_NAME=$(git config --global user.name 2>/dev/null || true); fi
if [ -z "$USER_NAME" ]; then USER_NAME=$(git config user.email 2>/dev/null | cut -d@ -f1 || true); fi
if [ -z "$USER_NAME" ]; then _u=$(whoami 2>/dev/null || true); if [[ "$_u" != "root" ]]; then USER_NAME="$_u"; fi; fi
if [ -z "$USER_NAME" ]; then USER_NAME="TCase user"; fi
```

### 第3步：代码生成（核心步骤——必须完成）

#### 3a. 判断用例类型

根据 `code` 字段内容判断类型：

| 类型 | 判断条件 | 你要做什么 |
|------|----------|------------|
| **高分用例** | `code` 中包含 `def test_`、`import`、`#TODO` 等 Python 代码结构 | 补全所有 `#TODO` 标记处的代码，使其成为完整可执行的测试 |
| **低分用例** | `code` 是文本列表，包含"用例名:"、"步骤1:"、"预期1:"等描述性文字 | **从零生成**完整的测试代码 |

**⚠️ 无论哪种类型，你都必须生成代码并调用 write_to_file / replace_in_file 写入文件！**

#### 3b. 高分用例处理

`code` 字段已经是带有 `#TODO` 标记的 Python 代码框架。你需要：
1. 将 `code` 作为代码骨架
2. 找到所有 `#TODO` 标记的位置
3. 根据 `order` 中的模板规范和上下文，补全每个 TODO 处的具体实现
4. 确保补全后的代码完整可执行

#### 3c. 纯文本用例处理

`code` 字段是纯文字的用例描述，格式如：
```
用例名: 
测试场景: 
步骤1: 
预期1: 
步骤2: 
预期2: 
```

你必须从零生成完整代码，具体步骤：

1. **解析用例描述** → 从 `code` 字段提取：用例名、测试场景、每个步骤的操作和预期结果
2. **搜索项目中的实现线索** → 用 `codebase_search` 或 `search_content` 在项目中搜索用例提到的业务动作、API 名称、资源名、字段名等直接线索，找到调用方式、参数格式、导入路径
3. **首次未命中时继续扩展搜索** → 继续搜索别名、同义动作、封装层、调用链等间接实现；同时查找相似测试封装
4. **读取项目中的类似测试** → 找到项目中已有的测试文件，理解项目的测试编写模式（如何初始化、如何调用接口、如何断言）
5. **按 `order` 模板生成完整代码** → 严格遵循 `order` 字段中的代码模板规范（包括文件头、import 格式、装饰器、函数签名、注释风格等），结合搜索到的 API 用法，生成完整的测试函数
6. **为每个步骤添加断言** → 每个测试步骤必须有对应的 assert 语句验证预期结果

**禁止**因为“未直接搜到同名 API”就回复无法生成；只要还能定位到相邻实现、封装入口、调用链节点或相似测试，就必须继续完成代码生成。

#### 3d. dir_append 模式的文件名构造

当工作流为 `dir_append` 时，MCP 返回的 `file_path` 是一个**目录路径**，你需要自行构造文件名：

1. 目标目录 = `order` 中的目标路径
2. 文件名提取优先级：
   - **优先**：代码中 `@File` 注释指定的文件名
   - **次选**：从测试函数名转换（如 `def test_auth_delete_user` → `test_auth_delete_user.py`）
3. 完整路径 = 目标目录 + "/" + 文件名

### 第4步：代码质量检查（写入前）

| 规则 | 说明 |
|------|------|
| 零 TODO | 输出禁止 TODO/FIXME/XXX |
| 零 pass | 禁止空函数体 |
| 完整实现 | 每个测试步骤必须有具体代码 |
| 导入完整 | 所有导入语句齐全 |
| author 已替换 | 使用真实用户名 |
| tcase_uuid 逐函数打标 | 同一个文件中每个 `test_` 函数必须在函数签名下方独立标注 `tcase_uuid`，严禁省略或共用；标注位置优先参考仓库已有的元信息方式（装饰器/docstring/注解），无先例则用注释 `# tcase_uuid: <值>`；已有非空且为标准 UUID 格式的值禁止修改（占位符除外） |

> **tcase_uuid 生成方式**：**优先复用 `design_case_uuid`**——如果该测试函数的 docstring/元信息中已有 `design_case_uuid`，则 `tcase_uuid` 直接取与 `design_case_uuid` 相同的值，无需调用生成脚本；仅当测试函数没有 `design_case_uuid` 时，才提取函数名，一次性调用 `python3 <skill_dir>/scripts/generate_tcase_uuid.py <repo> <branch> '["func_name_1", "func_name_2", ...]'`（数组长度不限，按实际函数数量传入）批量生成，返回 JSON 映射，逐个打标到对应函数。**建议在写入文件前先确定每个函数的 tcase_uuid 值，直接嵌入代码，避免使用占位符再替换。**

### 第5步：写入文件

根据工作流模式执行：

- **file_append**（追加到现有文件）：使用 `replace_in_file` 追加到文件末尾，**严禁修改现有代码**
- **create_file**（创建新文件）：使用 `write_to_file` 创建文件
- **dir_append**（目录下创建文件）：根据用例名生成文件名，使用 `write_to_file` 在目录下创建

### 第6步：写入验证（不可跳过！）

写入后**必须**使用 `read_file` 验证：

1. 文件是否存在
2. 文件内容是否非空
3. 文件是否包含预期的测试函数（`test_` 开头）
4. **代码格式是否符合 `order` 中的模板规范**
5. **所有 `tcase_uuid` 的值均为标准 RFC4122 UUID 格式**（`xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx`），非标准格式（如占位符）必须替换为真实 UUID 后才能通过验证

**验证失败处理**：直接使用 `write_to_file` 重新写入，写入后再次验证。最多重试 3 次。

### 验证检查清单

```
文件写入验证：
- [ ] 已读取目标文件
- [ ] 文件内容非空
- [ ] 文件包含 test_ 开头的函数
- [ ] 代码格式符合`MCP中order`模板规范
- [ ] 文件路径已记录（用于上传步骤）
```

## 示例执行流程

### 输入：MCP 返回内容

```json
{
  "order": "## TCase 任务\n**模式**: generate\n**工作流**: create_file\n**仓库**: xxx\n**路径**: xxx/yyy/test_zzz.py\n...",
  "code": "用例名: 某功能测试\n测试场景: 验证某功能\n步骤1: 调用某接口\n预期1: 返回成功",
  "status": "success"
}
```

### 你的执行步骤

```markdown
1. 解析 order：
2. 获取用户名
3. 搜索用例相关实现（直接线索 → 别名/封装层/调用链逐层扩展）
4. 读取相似测试和公共封装
5. 生成完整测试函数代码
6. 替换 author 为真实用户名
7. Write: 写入到已确认的目标路径（用户指定路径优先；未指定时使用 order 中的路径）
8. Read: 验证文件写入成功
9. 返回结果:
   - 文件路径: {实际写入路径}
   - 写入状态: 成功/失败
   - 验证状态: 已确认文件存在
```

**⚠️ 关键：文件路径必须遵循已确认的生成路径策略：用户指定路径优先；如果用户未指定，则使用 order 中的 `路径` 字段。**
