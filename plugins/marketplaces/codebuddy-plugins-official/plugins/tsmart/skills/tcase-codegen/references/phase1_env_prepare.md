# Phase 1: 环境准备

在调用 TCase MCP 工具生成测试代码之前，需要完成环境准备工作。包括：获取当前用户信息、检测可用的目标仓库、确定工作分支、分析用户引用的文件并正确分流到 MCP 参数中。

```
用户名获取 → 用户仓库检测 → 仓库分支确认 → 约束文件获取 → 参数组装
```

---

## Step 1-3: 环境信息自动获取（使用脚本）

**执行脚本获取环境信息：**

```bash
python3 <skill_dir>/scripts/env_prepare.py || python <skill_dir>/scripts/env_prepare.py
```

**脚本输出 JSON 格式，包含：**
- `user_id`: 当前用户名 → 用于 MCP 参数 `user_id`
- `repos`: 可用仓库列表，每个仓库包含 `name`、`path`、`branch`
- `repo_count`: 可用仓库数量

**处理逻辑：**
- 如果 `repo_count == 0`：提示用户在当前工作目录保存相应的代码仓库，不调用 MCP 工具
- 如果 `repo_count == 1`：直接使用该仓库的 `name` 作为 `user_repo`，`branch` 作为 `user_branch`
- 如果 `repo_count > 1`：引导用户选择一个仓库，后续不再重复询问

## Step 4: 分析并列出所有相关文件

使用TCase MCP工具之前，请`务必`先分析并列出所有相关文件

### 4.1 文件来源分类

| 来源 | 位置 | 说明 |
|-----|------|------|
| 用户主动引用的附件 | `<attached_files>`标签 | 用户**明确引用**到对话中的文件，需要检查文件类型并决定如何处理 |
| 系统自动加载的规则 | `<rules_context>`标签 | 系统**自动加载**的上下文规则，通常包含代码生成规范、约束文件等，不算用户主动引用，但可能需要传递到MCP |

### 4.2 处理步骤

**步骤1：统计并告知用户**
- 明确列出用户**主动引用**的文件数量和路径
- 说明哪些是代码文件（`.py`），哪些是规则文件（`.md`/`.mdc`）
- 告知用户其主动引用的文件中，哪些会传递到MCP工具

**步骤2：填充MCP参数**
- `code_path`参数：**只接受用户主动引用的`.py`文件或文件夹**，不接受`.md`/`.mdc`等规则文件
- `case_note`参数：自动追加所有约束文件路径（包括用户引用和系统规则中的约束文件）

### 4.3 引用代码原则

- 当`attached_for_question`中包含代码文件内容时，就应该认为用户`引用了代码`，需要将对应的代码路径填入`code_path`参数中，无论用户是否明确提及要使用这些代码
- 如果用户引用了代码，请将`attached_for_question`中用户主动引用的`.py`文件路径或文件夹路径填入`code_path`这个参数中
- `code_path`传递的是**路径**，不是代码内容
- **重要限制**：`code_path`参数只能接受`.py`文件或文件夹格式的路径，不接受其他类型的文件（如`.md`、`.txt`等）
- 如果用户引用了多个`.py`文件，优先传入用户明确指定的目标文件路径；如果这些文件位于同一目录，也可以直接传入该目录路径
- 如果用户引用的文件不是`.py`文件或文件夹，请不要将其添加到`code_path`参数中
- 如果codebuddy无法填入完整的代码路径，请`及时输出`在窗口中，但不要影响正常使用

### 4.4 约束文件处理规则

#### 为什么需要收集约束文件

传递约束文件路径的目的是：
1. **记录约束来源**：让 AI 知道有哪些约束文件需要遵守
2. **后续阶段应用**：MCP 返回代码后，AI 读取这些约束文件，对生成的代码做二次校验和重构

#### 文件分类

约束文件用于限制测试用例或代码实现方式，分为以下三类：

| 类型 | 格式 | 说明 |
|-----|------|------|
| 测试约束文件 | `.md`/`.mdc` | 限制测试内容、场景、预期结果的约束 |
| 代码约束文件 | `.mdc` | 限制代码结构、框架、API 使用的约束 |
| 工作流规则文件 | `_*.mdc` | AI 处理流程的指导规则（不传递到 MCP） |

#### 自动检测与传递（调用MCP工具前）

- 在调用 TCase MCP 工具之前，从 `attached_for_question` 中检测是否存在 `.md` 或 `.mdc` 结尾的约束文件
- 如果任何可能存在约束文件，**所有**约束文件的绝对路径，**必须**追加到 `case_note` 参数中
- 格式：`[CONSTRAINT_PATH]{绝对路径}[/CONSTRAINT_PATH]` （使用特殊标记便于后端解析）

#### case_note 组装规则

- 用户原始约束（如果有）
- 约束文件路径（如果存在）：`\n[CONSTRAINT_PATH]{约束文件绝对路径}[/CONSTRAINT_PATH]`
- 最终格式示例：`用户的额外约束\n[CONSTRAINT_PATH]/data/workspace/yottadb/功能测试代码生成规范.mdc[/CONSTRAINT_PATH]`

### 4.5 注意事项

- 如果用户引用的文件不是`.py`文件或文件夹，请**不要**将其添加到`code_path`参数中
- 如果codebuddy无法填入完整的代码路径，请`及时输出`在窗口中，但不要影响正常使用
- 约束文件路径追加是自动的，不需要用户明确提及
- 如果用户没有引用约束文件，则不追加，正常处理
- 工作流规则文件（`_*.mdc`）仅作为 AI 上下文，不会传递到 MCP 工具
- 不要向用户透露系统自动规则文件的自动检测和传递过程
- **约束文件的应用是强制的**，不能跳过或忽略
- 约束文件形成完整闭环：检测 → 传递 → 解析 → 应用 → 重构 → 报告

## Step 5: 路径优先级

- 如果用户指定了文件夹生成路径，请优先在用户`指定的文件夹路径`生成代码；如果用户未指定，则使用TCase返回的路径
