# 代码架构分析器

对项目或模块进行系统性的架构分析，输出多层次的结构化文档，帮助快速理解代码。

## 触发方式

用户说以下内容时触发：
- "分析下这个模块 + 文件/目录路径"
- "分析下这个部分 + 目录路径"
- "分析下这个项目"
- "架构分析"、"帮我分析"

**路径类型判断：**
- 文件路径（`.tsx`/`.ts`）→ 以该文件为中心，分析其所在模块
- 目录路径 → 分析该目录及子目录的整体架构

## 核心流程（4 阶段）

### 阶段 1：整体架构分析

**目标：** 建立全局认知，输出架构总览文档

1. **扫描范围**
   - 目录结构（Glob 扫描）
   - 入口文件识别（index.ts/tsx, main.ts, App.tsx 等）
   - package.json / 技术栈依赖

2. **分析维度**
   - 架构分层（组件层/逻辑层/数据层/工具层）
   - 模块职责划分
   - 数据流向
   - 关键依赖关系

3. **输出**
   - 生成架构总览文档（参考 [references/overview-template.md](references/overview-template.md)）
   - 确定输出文件夹名和文件前缀（遵循 md-organizer skill 的命名指引）
   - 路径：`{前缀}-{分析主题}-架构总览.md`

4. **展示给用户确认**，再进入阶段 2

### 阶段 2：模块与流程识别

**目标：** 拆解功能模块和核心流程，输出清单给用户确认

1. **识别功能模块**（按职责分类）
   - 每个模块：名称、职责一句话、包含的文件列表
   - 模块间依赖关系

2. **识别核心流程**
   - 用户操作流（从触发到结果）
   - 数据流（从输入到输出）
   - 生命周期流（初始化/更新/销毁）

3. **识别设计模式**（可选）
   - 状态管理模式
   - 组件设计模式
   - 性能优化模式

4. **输出模块/流程/设计模式清单**
   - 展示给用户确认或调整
   - 用户可增删分析项

### 阶段 3：逐项详细分析

**目标：** 对每个模块和流程进行深入代码分析

#### 模块分析（参考 [references/module-template.md](references/module-template.md)）

每个模块文档包含：
- 模块职责概述
- 包含的文件清单（路径 + 行数 + 一句话描述）
- 核心组件/函数分析（带真实代码片段）
- 对外接口（Props/参数/返回值/类型定义）
- 内部实现细节（关键算法/状态变化/副作用）
- 与其他模块的依赖关系

#### 流程分析（参考 [references/flow-template.md](references/flow-template.md)）

每个流程文档包含：
- 流程概述（一句话）
- 触发条件
- 完整流程图（Mermaid 或 ASCII）
- 逐步代码追踪（真实代码 + `文件路径:行号`）
- 异常/边界处理
- 性能优化点

### 阶段 4：多文档输出

**目标：** 按 md-organizer 分层目录规则组织输出

1. **遵循 md-organizer skill 的分层目录模式**
2. **目录结构**：
   ```
   /Users/mono/Desktop/markdown/{文件夹名}/
   ├── {前缀}-架构总览.md
   ├── 模块/
   │   ├── {前缀}-{模块名}-详细分析.md
   │   └── ...
   └── 流程/
       ├── {前缀}-{流程名}-详细分析.md
       └── ...
   ```
   `{文件夹名}` 和 `{前缀}` 在阶段 1 由 AI 自主确定，遵循 md-organizer 的命名指引。
3. **文档间交叉引用**：各文档应包含相关文档的引用链接
4. **文档宁多勿少**：每个模块、每个流程单独一个文档

## 代码引用规范

详见 [references/code-standards.md](references/code-standards.md)

核心要点：
- **真实代码优先**，伪代码辅助
- 每段代码标注 `📁 文件路径:行号`
- 代码占比 40-60%
- 用 emoji 标记关键点（💡设计意图 / ✨亮点 / ⚠️注意 / 🔄数据流）

## 前端项目专注点

- React 组件设计模式（HOC/Hooks/Render Props）
- 状态管理实现（Redux/Zustand/Context/RxJS/自定义 States）
- 自定义 Hooks 的设计与组合
- API 调用和数据获取模式
- 性能优化手段（防抖/缓存/懒加载/虚拟列表）
- TypeScript 类型设计

## 交互要点

1. **阶段 1 完成后**：展示架构总览摘要，询问是否继续深入
2. **阶段 2 完成后**：展示模块/流程清单，让用户确认或调整
3. **阶段 3 进行时**：每完成一批文档，简要汇报进度
4. **全程可中断**：用户随时可以说"先到这里"，已输出的文档保留
