# 代码架构分析器

以**流程为主线**对项目或模块进行系统性架构分析，输出多层次的结构化技术文档。

**核心理念：以流程为骨架，模块为血肉，图为主要表达手段。**

## 触发方式

用户说以下内容时触发：
- "分析下这个模块 + 文件/目录路径"
- "分析下这个部分 + 目录路径"
- "分析下这个项目"
- "架构分析"、"帮我分析"

**路径类型判断：**
- 文件路径（`.tsx`/`.ts`）→ 以该文件为中心，分析其所在模块
- 目录路径 → 分析该目录及子目录的整体架构

## 文档定位

这是**技术文档**，不是学习教程：
- 不需要学习路径、推荐阅读顺序、学习优先级
- 不需要索引速查、阅读建议
- 流程分析到位，读者自然知道该深入哪里
- 图表优先于代码 — 先用图讲清楚流程和关系，再用精简代码辅助说明

## 核心流程（4 阶段）

### 阶段 1：全景架构

**目标：** 建立全局认知，输出架构总览文档

1. **扫描范围**
   - 目录结构（Glob 扫描）
   - 入口文件识别（index.ts/tsx, main.ts, App.tsx 等）
   - package.json / 技术栈依赖

2. **分析重点**
   - 这个项目/模块做什么（一句话定位）
   - 整体运行流程分几大步骤（全景流程图）
   - 模块之间是什么关系（模块关系图）
   - 有哪些核心流程（流程清单）

3. **输出**
   - 生成架构总览文档（参考 [references/overview-template.md](references/overview-template.md)）
   - 确定输出文件夹名和文件前缀（遵循 md-organizer skill 的命名指引）
   - 路径：`{前缀}-架构总览.md`

4. **展示给用户确认**，再进入阶段 2

### 阶段 2：流程拆解与模块映射

**目标：** 以流程为线索拆解系统，识别核心流程并映射到模块

1. **识别核心流程**（2-5 条）
   - 用户操作流（从触发到结果）
   - 数据流（从输入到输出）
   - 生命周期流（初始化/更新/销毁）

2. **流程-模块映射**
   - 每条流程标注涉及哪些模块
   - 模块不独立列出，而是"挂载"在流程上 — 通过流程认识模块

3. **输出流程清单给用户确认**
   - 每条流程：名称、一句话描述、涉及模块列表
   - 用户可增删调整

### 阶段 3：逐流程深入分析

**目标：** 以流程为单位生成详细分析文档

每个流程生成一个独立文档（参考 [references/flow-template.md](references/flow-template.md)）：

#### 流程文档包含：

1. **流程概述** — 做什么、何时触发、涉及哪些模块
2. **参与者交互图**（Mermaid sequenceDiagram）— 模块间的消息传递和调用关系
3. **逐步分析** — 每步用文字说清楚做什么，辅以精简代码/伪代码 + `文件路径:行号`
4. **涉及模块速览**（参考 [references/module-template.md](references/module-template.md)）— 在流程上下文中介绍每个模块的职责和关键接口
5. **状态变化图**（Mermaid stateDiagram-v2）— 如果流程涉及状态变化
6. **数据流转图**（Mermaid flowchart）— 数据从输入经过各步骤变换到输出
7. **异常与边界处理** — 错误场景和处理方式
8. **关键设计决策** — 为什么这样实现，有什么取舍

### 阶段 4：多文档输出

**目标：** 按 md-organizer 规则组织输出

1. **遵循 md-organizer skill 的命名指引**
2. **目录结构**（按流程组织，不按模块/流程分开）：
   ```
   /Users/mono/Desktop/markdown/{文件夹名}/
   ├── {前缀}-架构总览.md              # 阶段 1 输出
   ├── {前缀}-{流程1名}-流程分析.md    # 阶段 3 输出
   ├── {前缀}-{流程2名}-流程分析.md
   └── {前缀}-{流程3名}-流程分析.md
   ```
   `{文件夹名}` 和 `{前缀}` 在阶段 1 由 AI 自主确定。
3. **文档间交叉引用**：各文档应包含相关文档的引用链接

## 图表与代码规范

详见 [references/code-standards.md](references/code-standards.md)

### 核心原则

**图表优先，代码辅助：**
- 每个文档至少包含 **2 种不同类型** 的 Mermaid 图
- 先用图讲清楚流程、关系、状态变化
- 代码是辅助说明手段，不是主体
- 代码可以是真实代码片段、伪代码、简化代码 — 目标是说清楚逻辑
- 每段代码标注 `文件路径:行号`（如果引用真实代码）

### 图表类型要求

| 文档类型 | 必选图 | 可选图 |
|---------|--------|--------|
| 架构总览 | flowchart TD（全景流程）+ flowchart LR（模块关系） | — |
| 流程分析 | sequenceDiagram（模块交互） | stateDiagram-v2（状态变化）、flowchart（数据流转） |

### Mermaid 语法安全规则

- Node labels 必须加引号：`A["CLI Entry"]` 而不是 `A[CLI Entry]`
- Node ID 只用英文字母数字：`CoreModule` 而不是 `CoreModule.123`
- subgraph ID 用英文，不和 node ID 冲突
- 内嵌引号用 HTML entity：`A["Config &quot;key&quot;"]`
- 节点数 > 6 时用 subgraph 分组
- 节点数 > 20 时拆成多个图

## 前端项目专注点

- React 组件设计模式（HOC/Hooks/Render Props）
- 状态管理实现（Redux/Zustand/Context/RxJS/自定义 States）
- 自定义 Hooks 的设计与组合
- API 调用和数据获取模式
- 事件系统与跨组件通信
- TypeScript 类型设计

## 交互要点

1. **阶段 1 完成后**：展示架构总览摘要，询问是否继续深入
2. **阶段 2 完成后**：展示流程清单，让用户确认或调整
3. **阶段 3 进行时**：每完成一个流程文档，简要汇报进度
4. **全程可中断**：用户随时可以说"先到这里"，已输出的文档保留
